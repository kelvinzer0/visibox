     7→     7→## 0. RINGKASAN EKSEKUTIF
     8→     8→
     9→     9→VisiBox adalah fork bash yang menambahkan 5 kapabilitas untuk dikendalikan AI agent lewat protokol JSON:
    10→    10→
    11→    11→1. **ID system** — setiap operasi punya `request_id` (milik caller) dan `response_id` (milik VisiBox), untuk audit dan referensi silang.
    12→    12→2. **Interactive session control** — AI bisa spawn proses interaktif (ssh, python, mysql), kirim input, baca output, kapan saja, tanpa proses itu mati di antara request.
    13→    13→3. **Output pagination** — output besar dipotong per halaman dengan cursor, supaya AI tidak overload context window.
    14→    14→4. **Line numbers** — setiap baris output shell dilengkapi nomor baris (seperti code editor), memudahkan AI merujuk baris spesifik saat analisis output.
    15→    15→5. **Keyword-based pagination jump** — AI bisa mencari keyword di seluruh output dan langsung loncat ke halaman pagination yang mengandung keyword tersebut, tanpa harus scroll manual halaman per halaman.
    16→    16→
    17→    17→Dokumen ini adalah **versi v3** yang membangun di atas v2 (rewrite). Enam masalah desain di draft v1 sudah diperbaiki di v2 — lihat §1 untuk daftar lengkap apa yang berubah dari v1. Fitur baru v3 (#4 dan #5) dijelaskan di §1.3.

    22→    22→
    23→    23→### 1.1 Perubahan v1 → v2
    24→    24→
    25→    25→| # | Masalah di v1 | Perbaikan di v2 |
    26→    26→|---|---|---|
    27→    27→| 1 | `execute` fork ulang shell baru per command → `cd`, `export` tidak persist antar request | Command non-session dieksekusi di **proses bash induk yang sama**, lewat jalur eksekusi asli bash. Shell state (cwd, env, alias) persist secara native, sama seperti shell interaktif biasa. |
    28→    28→| 2 | Truncation tetap drain seluruh output dari pipe → tidak menghemat compute, hanya menghemat token | Didokumentasikan eksplisit: `output_limit` adalah **proteksi token**, bukan proteksi compute. Ditambahkan rekomendasi command-level limiting (`head`, `LIMIT`) sebagai tanggung jawab AI agent, bukan VisiBox. |
    29→    29→| 3 | Response history ring buffer tanpa kebijakan retensi/limit memori | Ring buffer punya **kapasitas eksplisit** (jumlah entri + total bytes), kebijakan eviction (FIFO), dan response error standar (`ERR_RESPONSE_EXPIRED`) kalau `fetch_page` mengacu ke response yang sudah di-evict. |
    30→    30→| 4 | PTY read loop pakai busy-poll `usleep(10000)` per session → tidak scale ke concurrent session | Diganti `poll()`/`epoll` dari Fase 2 (bukan ditunda ke Fase 3). Satu event loop memantau semua PTY fd sekaligus. Tidak ada busy-wait. |
    31→    31→| 5 | Prompt detection regex rapuh terhadap ANSI escape code | Output PTY di-strip ANSI **sebelum** regex match (raw output tetap disimpan untuk ditampilkan ke AI). |
    32→    32→| 6 | Tidak jelas kapan AI harus pakai `execute` vs `session_start` | Ditambahkan aturan keputusan eksplisit di §3, dengan tabel kapan pakai yang mana. |

    35→    35→
    36→    36→| # | Fitur Baru v3 | Deskripsi |
    37→    37→|---|---|---|
    38→    38→| 7 | **Line numbers di output** | Setiap baris output shell diberi nomor baris absolut (berdasarkan posisi di keseluruhan output, bukan per-halaman). AI bisa merujuk "baris 42" dan langsung tahu itu baris ke-42 dari total output. Nomor baris konsisten di semua halaman — halaman 2 tidak mulai dari 1 lagi, melainkan dari `page_size + 1`. |

Menghilangkan kebutuhan untuk scroll halaman per halaman saat mencari sesuatu di output besar. |

    43→    43→## 2. PRINSIP DESAIN
    44→    44→
    45→    45→Ini bukan sekadar daftar fitur — ini aturan yang dipegang konsisten di seluruh implementasi, supaya saat coding solo nanti tidak bikin keputusan ad-hoc yang saling kontradiksi.
    46→    46→
    47→    47→**P1 — Shell state itu sakral.**
    48→    48→Command non-interaktif (`execute`) HARUS berjalan di proses bash yang sama, bukan subprocess terpisah. Kalau ini dilanggar, seluruh value proposition "AI bisa `cd` lalu `ls`" hilang.
    49→    49→
    50→    50→**P2 — Pagination melindungi token, bukan compute.**
    51→    51→VisiBox tidak menjanjikan command berhenti lebih cepat karena `output_limit` kecil. Itu tanggung jawab command itu sendiri (`head`, `LIMIT N`, dll). Dokumentasikan ini ke pengguna API supaya ekspektasi benar.
    52→    52→
    53→    53→**P3 — Tidak ada busy-wait di hot path.**
    54→    54→Semua I/O menunggu (PTY read, socket read) lewat `poll()`/`epoll`, bukan sleep-loop. Ini bukan optimisasi prematur — ini syarat supaya Fase 2 tidak perlu rewrite di Fase 3.
    55→    55→
    56→    56→**P4 — Semua resource punya batas eksplisit dan kebijakan eviction.**
    57→    57→Ring buffer response, jumlah session aktif, ukuran output buffer per session — semua punya angka konkret di config, bukan "in-memory, unbounded" yang baru ditemukan masalahnya saat production.
    58→    58→
    59→    59→**P5 — Ambiguitas protokol = bug spec, bukan bug implementasi.**
    60→    60→Kalau ada pertanyaan "request macam apa yang harus dikirim AI di situasi X" yang tidak terjawab oleh dokumen ini, itu dianggap PRD belum lengkap — bukan sesuatu yang "akan jelas saat coding". Setiap request type punya kontrak lengkap di §5.
    61→    61→
    62→    62→**P6 — Nomor baris itu absolut dan konsisten.** *(BARU v3)*
    63→    63→Line numbers merepresentasikan posisi baris di keseluruhan output, bukan posisi relatif per halaman. Halaman 1 = baris 1–100, halaman 2 = baris 101–200, dst. Ini memungkinkan AI merujuk baris spesifik secara unik tanpa ambigu "halaman berapa, baris ke berapa".
    64→    64→
    65→    65→**P7 — Pencarian harus cepat tanpa fetch seluruh output.** *(BARU v3)*
    66→    66→`search_jump` harus bisa mencari keyword tanpa mengirim seluruh output ke AI. Implementasi pencarian dilakukan server-side pada buffered output, bukan client-side. AI hanya menerima halaman yang relevan.

    69→    69→
    70→    70→## 3. KAPAN PAKAI `execute` VS `session_start`
    71→    71→
    72→    72→Ini keputusan yang sering diabaikan di desain serupa, padahal krusial buat AI agent supaya tidak salah pilih jalur.
    73→    73→
    74→    74→| Situasi | Jalur | Alasan |
    75→    75→|---|---|---|
    76→    76→| Command selesai dengan sendirinya, exit code jelas (`ls`, `df -h`, `git status`) | `execute` | Tidak butuh state interaktif lintas-command selain shell env (yang otomatis persist via P1). |
    77→    77→| Command butuh input interaktif (password, konfirmasi `y/n`, REPL) | `session_start` + `session_input` | `execute` tidak punya jalur untuk mengirim input setelah command jalan. |
    78→    78→| Command yang sengaja tidak pernah exit (`tail -f`, server foreground, `watch`) | `session_start` | `execute` akan menunggu exit yang tidak pernah terjadi → harus timeout-kill, yang merusak semantik "command selesai, ini hasilnya". |
    79→    79→| Butuh banyak command berurutan dengan koneksi/proses yang sama (ssh lalu beberapa command remote) | `session_start` sekali, lalu beberapa `session_input` | Membuka ssh baru per command = re-auth tiap kali, tidak efisien, dan kalau session punya state (mysql `USE db;`) akan hilang. |
    80→    80→| Command lokal sederhana tapi dipanggil bertubi-tubi dengan dependency pada `cwd`/env yang diubah command sebelumnya | `execute` (berkat P1, ini AMAN — cwd persist otomatis) | Tidak perlu session untuk ini; ini justru kasus yang dulu rusak di v1. |
    81→    81→
    82→    82→Aturan ringkas untuk AI agent: **kalau command itu sendiri akan exit dan tidak butuh input tambahan, pakai `execute`. Selain itu, `session_start`.**

   132→   132→
   133→   133→### 4.2 Bagaimana `execute` TIDAK fork shell baru (perbaikan masalah #1)
   134→   134→
   135→   135→Ini bagian paling penting yang berubah dari v1. Penjelasan mekanismenya:
   136→   136→
   137→   137→**Yang SALAH (v1):** parent fork() → child jalankan command via `visibox_execute_original()` di proses terpisah → exit. Command seperti `cd` cuma mengubah cwd di child, hilang begitu child exit. Parent (proses VisiBox utama) cwd-nya tidak pernah berubah.
   138→   138→
   139→   139→**Yang BENAR (v2+):** Hook ditempatkan di `eval.c`, **setelah** bash parse command line jadi command struct, tapi **sebelum/sesudah** bash menjalankannya lewat jalur normal (`execute_command()` bash asli). Untuk command yang sudah `fork` secara native oleh bash (kebanyakan command eksternal memang fork di bash), kita tidak menambah fork lagi — kita hanya:
   140→   140→
   141→   141→1. Redirect fd 1/2 proses VisiBox **sementara** (lewat `dup2` + simpan fd asli) sebelum memanggil jalur eksekusi bash asli.
   142→   142→2. Panggil `execute_command()` bash asli — bash sendiri yang menentukan apakah perlu fork (command eksternal) atau tidak (builtin seperti `cd`, `export`).
   143→   143→3. Setelah command selesai, restore fd 1/2 ke semula.
   144→   144→4. Buffer yang menampung output adalah pipe yang dibaca di proses VisiBox utama sendiri (bukan child terpisah) — karena fd 1/2 sudah diarahkan ke pipe itu selama command jalan.
   145→   145→
   146→   146→Konsekuensi: builtin (`cd`, `export`, `alias`, `unset`) memodifikasi state proses VisiBox utama secara langsung, persis seperti shell interaktif normal. Command eksternal tetap fork (seperti biasanya di bash), tapi itu fork yang sama yang sudah dilakukan bash secara native — bukan fork tambahan dari VisiBox.
   147→   147→
   148→   148→```c
   149→   149→// execute_cmd.c — MODIFIED (v2+, bukan v1)
   150→   150→int visibox_execute_and_capture(VisiboxRequest *req, VisiboxResponse *res) {
   151→   151→    visibox_generate_id(res->response_id, "res_");
   152→   152→
   153→   153→    int pipefd[2];
   154→   154→    pipe(pipefd);
   155→   155→
   156→   156→    int saved_stdout = dup(STDOUT_FILENO);
   157→   157→    int saved_stderr = dup(STDERR_FILENO);
   158→   158→    dup2(pipefd[1], STDOUT_FILENO);
   159→   159→    dup2(pipefd[1], STDERR_FILENO);
   160→   160→    close(pipefd[1]);
   161→   161→
   162→   162→    // PENTING: set pipe read-end non-blocking SEBELUM eksekusi,
   163→   163→    // supaya kita bisa membaca progresif sambil command masih jalan
   164→   164→    // (perlu untuk early-truncation tanpa deadlock pada command yang
   165→   165→    // outputnya lebih besar dari pipe buffer kernel, biasanya 64KB)
   166→   166→    fcntl(pipefd[0], F_SETFL, O_NONBLOCK);
   167→   167→
   168→   168→    struct timespec start, end;
   169→   169→    clock_gettime(CLOCK_MONOTONIC, &start);
   170→   170→
   171→   171→    // INI jalur eksekusi BASH ASLI — bukan fork+exec buatan VisiBox.
   172→   172→    // Builtin (cd, export, dll) jalan langsung di proses ini.
   173→   173→    // Command eksternal: bash sendiri yang fork, seperti perilaku normal.
   174→   174→    int exit_code = execute_command(req->parsed_command);  // fungsi bash native
   175→   175→
   176→   176→    clock_gettime(CLOCK_MONOTONIC, &end);
   177→   177→
   178→   178→    // Restore fd SEGERA setelah command selesai — sebelum baca pipe,
   179→   179→    // supaya VisiBox bisa nulis log/error ke stdout asli kalau perlu
   180→   180→    dup2(saved_stdout, STDOUT_FILENO);
   181→   181→    dup2(saved_stderr, STDERR_FILENO);
   182→   182→    close(saved_stdout);
   183→   183→    close(saved_stderr);
   184→   184→
   185→   185→    OutputBuffer *buf = visibox_output_buffer_new();
   186→   186→    visibox_drain_pipe_with_limit(pipefd[0], buf, req->output_limit, req->page_mode);
   187→   187→    close(pipefd[0]);
   188→   188→
   189→   189→    res->exit_code = exit_code;
   190→   190→    res->duration_ms = timespec_diff_ms(&start, &end);
   191→   191→    res->output = visibox_output_buffer_get_page(buf, 1, req->output_limit, req->page_mode);
   192→   192→    res->output_lines = buf->total_lines;
   193→   193→    res->output_bytes = buf->total_bytes;
   194→   194→    res->output_truncated = buf->truncated;
   195→   195→
   196→   196→    // [BARU v3] Format line numbers jika diminta
   197→   197→    if (req->options.line_numbers) {
   198→   198→        visibox_linenums_inject(res->output, 1, req->output_limit, buf->total_lines);
   199→   199→        res->line_numbers = true;
   200→   200→        res->line_start = 1;
   201→   201→        res->line_end = visibox_min(req->output_limit, buf->total_lines);
   202→   202→    }
   203→   203→
   204→   204→    if (buf->truncated) {
   205→   205→        visibox_paginator_build_metadata(buf, req, res);
   206→   206→    }
   207→   207→
   208→   208→    visibox_response_store_add(res, buf);  // simpan buf untuk fetch_page/search_jump nanti
   209→   209→    return 0;
   210→   210→}
   211→   211→```
   212→   212→
   213→   213→**Catatan implementasi penting:** `execute_command()` adalah fungsi internal bash (nama sebenarnya bisa berbeda tergantung versi source — cek `execute_cmd.c` upstream bash, biasanya `execute_command()` atau `execute_command_internal()`). Tugas pertama di Fase 1 adalah memetakan fungsi mana persisnya di source yang dipakai, dan memverifikasi lewat eksperimen kecil bahwa `cd` di dalam pemanggilan ini benar-benar mengubah `getcwd()` proses, bukan cuma child.
   214→   214→
   215→   215→### 4.3 Drain dengan limit — kebijakan diperjelas (perbaikan masalah #2)
   216→   216→
   217→   217→```c
   218→   218→// visibox_paginator.c
   219→   219→void visibox_drain_pipe_with_limit(int fd, OutputBuffer *buf,
   220→   220→                                     size_t limit, PaginationMode mode) {
   221→   221→    char chunk[4096];
   222→   222→    ssize_t n;
   223→   223→    bool limit_hit = false;
   224→   224→
   225→   225→    while (1) {
   226→   226→        n = read(fd, chunk, sizeof(chunk));
   227→   227→        if (n > 0) {
   228→   228→            if (!limit_hit) {
   229→   229→                visibox_output_buffer_append(buf, chunk, n);
   230→   230→                if (limit > 0 &&
   231→   231→                    ((mode == VISIBOX_PAGE_LINES && buf->total_lines >= limit) ||
   232→   232→                     (mode == VISIBOX_PAGE_BYTES && buf->total_bytes >= limit))) {
   233→   233→                    buf->truncated = true;
   234→   234→                    limit_hit = true;
   235→   235→                    // CATATAN: kita TERUS membaca fd sampai EOF untuk mencegah
   236→   236→                    // proses penulis (command) blok menulis ke pipe penuh,
   237→   237→                    // tapi BERHENTI menyimpan ke buffer (buang chunk).
   238→   238→                    // Ini TIDAK menghemat waktu eksekusi command — command
   239→   239→                    // tetap jalan sampai selesai. Ini HANYA mencegah AI
   240→   240→                    // menerima output raksasa. Lihat P2 di §2.
   241→   241→                }
   242→   242→            }
   243→   243→            // kalau limit_hit, chunk dibuang — tapi total_lines/total_bytes
   244→   244→            // TETAP dihitung di luar buffer kalau include_total=true di config,
   245→   245→            // supaya pagination metadata (total_lines) akurat
   246→   246→            if (limit_hit) {
   247→   247→                visibox_output_buffer_count_only(buf, chunk, n);
   248→   248→            }
   249→   249→        } else if (n == 0) {
   250→   250→            break;  // EOF, command selesai menulis
   251→   251→        } else {
   252→   252→            if (errno == EAGAIN || errno == EWOULDBLOCK) {
   253→   253→                struct pollfd pfd = { .fd = fd, .events = POLLIN };
   254→   254→                poll(&pfd, 1, -1);  // tunggu data tanpa busy-wait
   255→   255→                continue;
   256→   256→            }
   257→   257→            break;  // error lain, stop
   258→   258→        }
   259→   259→    }
   260→   260→}
   261→   261→```

   265→   265→## 5. FITUR BARU v3 — DETAIL TEKNIS
   266→   266→
   267→   267→### 5A. LINE NUMBERS DI OUTPUT
   268→   268→
   269→   269→#### 5A.1 Konsep
   270→   270→
   271→   271→Setiap baris output shell yang dikembalikan ke AI bisa dilengkapi nomor barit absolut. Nomor baris ini merepresentasikan posisi baris di **keseluruhan output** command, bukan posisi relatif di halaman saat ini.
   272→   272→
   273→   273→**Contoh output `ls -la` dengan 100 file, page_size=10, line_numbers=true:**
   274→   274→
   275→   275→Halaman 1:
   276→   276→```
   277→   277→  1 │ total 504
   278→   278→  2 │ drwxr-xr-x  2 root root  4096 Jun 25 10:00 .
   279→   279→  3 │ drwxr-xr-x  3 root root  4096 Jun 25 09:55 ..
   280→   280→  4 │ -rw-r--r--  1 root root  8192 Jun 25 10:00 file1.txt
   281→   281→  5 │ -rw-r--r--  1 root root  4096 Jun 25 10:00 file2.txt
   282→   282→  6 │ -rw-r--r--  1 root root  4096 Jun 25 10:00 file3.txt
   283→   283→  7 │ -rw-r--r--  1 root root  4096 Jun 25 10:00 file4.txt
   284→   284→  8 │ -rw-r--r--  1 root root  4096 Jun 25 10:00 file5.txt
   285→   285→  9 │ -rw-r--r--  1 root root  4096 Jun 25 10:00 file6.txt
   286→   286→ 10 │ -rw-r--r--  1 root root  4096 Jun 25 10:00 file7.txt
   287→   287→```
   288→   288→
   289→   289→Halaman 2 (`fetch_page` page=2):
   290→   290→```
   291→   291→ 11 │ -rw-r--r--  1 root root  4096 Jun 25 10:00 file8.txt
   292→   292→ 12 │ -rw-r--r--  1 root root  4096 Jun 25 10:00 file9.txt
   293→   293→ 13 │ -rw-r--r--  1 root root  4096 Jun 25 10:00 file10.txt
   294→   294→ ...
   295→   295→ 20 │ -rw-r--r--  1 root root  4096 Jun 25 10:00 file17.txt
   296→   296→```
   297→   297→
   298→   298→**Key insight:** Baris 1-10 di halaman 1, baris 11-20 di halaman 2. AI bisa merujuk "baris 15" dan itu unik di seluruh output — tidak ambigu "halaman 2 baris 5".
   299→   299→
   300→   300→#### 5A.2 Format Line Number
   301→   301→
   302→   302→```
   303→   303→{right-aligned number, width = digit_count(total_lines)} │ {content}
   304→   304→```
   305→   305→
   306→   306→- Pemisah default: ` │ ` (space, box-drawing vertical line U+2502, space)
   307→   307→- Lebar kolom nomor barit otomatis menyesuaikan jumlah digit total_lines
   308→   308→  - 1-9 baris → width 1
   309→   309→  - 10-99 baris → width 2
   310→   310→  - 100-999 baris → width 3
   311→   311→  - dst.
   312→   312→- Right-aligned, padded dengan spasi
   313→   313→- Baris kosong juga dapat nomor baris (kontennya string kosong setelah separator)
   314→   314→
   315→   315→#### 5A.3 Penggunaan di Protokol
   316→   316→
   317→   317→Line numbers dikontrol via field `options.line_numbers` (boolean, default: `false`).
   318→   318→
   319→   319→**`execute` dengan line numbers:**
   320→   320→```json
   321→   321→{
   322→   322→  "request_id": "req_001",
   323→   323→  "type": "execute",
   324→   324→  "command": "find /var/log -name '*.log' -type f",
   325→   325→  "options": {
   326→   326→    "output_limit": 50,
   327→   327→    "output_unit": "lines",
   328→   328→    "line_numbers": true
   329→   329→  }
   330→   330→}

   332→   332→
   333→   333→**Response:**

   340→   340→  "output": "  1 │ /var/log/syslog\n  2 │ /var/log/auth.log\n  3 │ /var/log/kern.log\n...",
   341→   341→  "output_lines": 127,

   381→   381→
   382→   382→#### 5A.4 Implementasi: `visibox_linenums.c`
   383→   383→
   384→   384→```c
   385→   385→// visibox_linenums.c
   386→   386→
   387→   387→// Inject line numbers ke string output yang sudah jadi.
   388→   388→// `line_start` = nomor baris pertama di string ini (absolut, bukan 0-based)
   389→   389→// `total_lines` = total baris di keseluruhan output (untuk hitung width)
   390→   390→// Mengembalikan string baru yang dialokasikan, caller harus free().
   391→   391→char *visibox_linenums_inject(const char *raw_output, size_t line_start,
   392→   392→                               size_t page_lines, size_t total_lines);
   393→   393→
   394→   394→// Helper: hitung jumlah digit yang diperlukan
   395→   395→int visibox_linenums_width(size_t total_lines) {
   396→   396→    if (total_lines == 0) return 1;
   397→   397→    int w = 0;
   398→   398→    size_t n = total_lines;
   399→   399→    while (n > 0) { w++; n /= 10; }
   400→   400→    return w;
   401→   401→}
   402→   402→
   403→   403→// Helper: format satu baris dengan line number
   404→   404→// Format: {num, right-aligned, width=w} │ {content}
   405→   405→// Memakai separator dari config (default: " │ ")
   406→   406→void visibox_linenums_format_line(char *out, size_t out_size,
   407→   407→                                   size_t line_num, int width,
   408→   408→                                   const char *separator,
   409→   409→                                   const char *content, size_t content_len);
   410→   410→```
   411→   411→
   412→   412→**Performa:** Line number injection dilakukan **setelah** output diambil dari buffer, bukan saat buffering. Ini berarti:
   413→   413→- Buffer tetap menyimpan raw output (tanpa line numbers) — hemat memori
   414→   414→- Line numbers dihitung on-the-fly saat response dibuat
   415→   415→- `search_jump` bisa mencari di raw buffer tanpa skip line numbers
   416→   416→- Format line numbers bisa berbeda antar request (satu client minta line numbers, client lain tidak — output yang sama dari buffer)
   417→   417→
   418→   418→#### 5A.5 Edge Cases
   419→   419→
   420→   420→| Kasus | Perilaku |
   421→   421→|---|---|
   422→   422→| Output kosong (0 baris) | `line_numbers: true` tapi output string kosong. `line_start: 0`, `line_end: 0`. |
   423→   423→| Baris terakhir tidak memiliki newline | Tetap diberi nomor baris. Line number mengikuti jumlah baris, bukan jumlah newline. |
   424→   424→| Output hanya 1 baris | Width = 1, format: `1 │ content` |
   425→   425→| Output 1000+ baris | Width = 4, format: `   1 │ content` ... `1000 │ content` |
   426→   426→| ANSI escape codes di output | Line numbers diletakkan SEBELUM ANSI codes per baris. Format: `  1 │ \e[31mred text\e[0m` — supaya AI tetap bisa parse ANSI. |
   427→   427→| `search_jump` + `line_numbers` | Output dari search_jump juga mendukung line_numbers=true, dan nomor barisnya tetap absolut. |

   430→   430→
   431→   431→### 5B. KEYWORD-BASED PAGINATION JUMP (`search_jump`)
   432→   432→
   433→   433→#### 5B.1 Konsep
   434→   434→
   435→   435→Saat output command sangat besar (misalnya 10.000 baris log), AI sering perlu mencari sesuatu spesifik (error message, IP address, nama file). Tanpa `search_jump`, AI harus:
   436→   436→1. Fetch halaman 1, scan manual
   437→   437→2. Fetch halaman 2, scan manual
   438→   438→3. ... berulang sampai ketemu
   439→   439→
   440→   440→`search_jump` menghilangkan ini dengan satu request: AI kirim keyword, VisiBox cari di seluruh buffered output, dan kembalikan halaman yang mengandung keyword beserta konteks.
   441→   441→
   442→   442→#### 5B.2 Request: `search_jump`
   443→   443→
   444→   444→```json
   445→   445→{
   446→   446→  "request_id": "req_060",
   447→   447→  "type": "search_jump",
   448→   448→  "response_id": "res_xxxxxxxx",
   449→   449→  "keyword": "ERROR",
   450→   450→  "options": {
   451→   451→    "case_sensitive": false,
   452→   452→    "occurrence": 1,
   453→   453→    "context_lines": 3,
   454→   454→    "output_limit": 50,
   455→   455→    "line_numbers": true
   456→   456→  }
   457→   457→}
   458→   458→```
   459→   459→
   460→   460→| Field | Wajib | Default | Catatan |
   461→   461→|---|---|---|---|
   462→   462→| `response_id` | ya | — | Response ID output yang ingin dicari (harus masih ada di ring buffer) |
   463→   463→| `keyword` | ya | — | String yang dicari. Bisa mengandung spasi. Tidak support regex di v3 (plaintext substring match). |
   464→   464→| `options.case_sensitive` | tidak | `false` | Pencarian case-insensitive secara default |
   465→   465→| `options.occurrence` | tidak | `1` | Kejadian ke-berapa yang diinginkan. `1` = kejadian pertama, `2` = kedua, dst. `-1` = kejadian terakhir. |
   466→   466→| `options.context_lines` | tidak | `3` | Jumlah baris konteks sebelum dan sesudah baris yang mengandung keyword. Total baris di output = 1 (baris keyword) + 2 × context_lines. |
   467→   467→| `options.output_limit` | tidak | dari config `pagination.default_page_size` | Batas maksimum baris yang dikembalikan. Jika context terlalu lebar, dipotong. |
   468→   468→| `options.line_numbers` | tidak | `false` | Sama seperti line numbers di `execute`/`fetch_page` (§5A) |
   469→   469→
   470→   470→#### 5B.3 Response: `search_jump_result`
   471→   471→
   472→   472→```json
   473→   473→{
   474→   474→  "request_id": "req_060",
   475→   475→  "response_id": "res_xxxxxxxx",
   476→   476→  "type": "search_jump_result",
   477→   477→  "keyword": "ERROR",
   478→   478→  "occurrence": 1,
   479→   479→  "total_occurrences": 47,
   480→   480→  "found": true,
   481→   481→  "found_line": 1842,
   482→   482→  "line_numbers": true,
   483→   483→  "line_start": 1839,
   484→   484→  "line_end": 1845,
   485→   485→  "output": "1839 │ [INFO] Starting worker process 3\n1840 │ [INFO] Connected to database\n1841 │ [WARN] High memory usage detected\n1842 │ [ERROR] Connection timeout to replica-3\n1843 │ [INFO] Retrying connection (attempt 1/3)\n1844 │ [INFO] Connected to replica-3\n1845 │ [INFO] Worker process 3 ready",
   486→   486→  "output_truncated": false,
   487→   487→  "page_hint": 37,
   488→   488→  "cursor": "cur_page_37_start"
   489→   489→}
   490→   490→```
   491→   491→
   492→   492→| Field | Arti |
   493→   493→|---|---|
   494→   494→| `found` | `true` jika keyword ditemukan, `false` jika tidak ada kejadian. |
   495→   495→| `total_occurrences` | Total jumlah baris yang mengandung keyword di seluruh output. Berguna buat AI menilai seberapa sering error muncul. |
   496→   496→| `found_line` | Nomor baris absolut (1-based) di mana keyword ditemukan pada occurrence yang diminta. |
   497→   497→| `line_start` / `line_end` | Range baris yang dikembalikan (found_line ± context_lines, dibatasi output_limit). |
   498→   498→| `output` | Baris-baris konteks dengan keyword di tengahnya. Jika `line_numbers: true`, diformat sesuai §5A. |
   499→   499→| `page_hint` | Nomor halaman yang mengandung `found_line` (berdasarkan page_size). AI bisa langsung `fetch_page` ke halaman ini kalau butuh lebih banyak konteks. |
   500→   500→| `cursor` | Cursor ke awal halaman `page_hint`. AI bisa pakai ini di `fetch_page` untuk langsung loncat. |
   501→   501→
   502→   502→#### 5B.4 Kasus `found: false`
   503→   503→
   504→   504→```json
   505→   505→{
   506→   506→  "request_id": "req_061",
   507→   507→  "response_id": "res_xxxxxxxx",
   508→   508→  "type": "search_jump_result",
   509→   509→  "keyword": "CRITICAL",
   510→   510→  "occurrence": 1,
   511→   511→  "total_occurrences": 0,
   512→   512→  "found": false,
   513→   513→  "output": "",
   514→   514→  "output_truncated": false
   515→   515→}
   516→   516→```
   517→   517→
   518→   518→Tidak ada error — ini bukan error, keyword memang tidak ada di output. AI bisa lanjut coba keyword lain.
   519→   519→
   520→   520→#### 5B.5 Kasus occurrence melebihi total_occurrences
   521→   521→
   522→   522→```json
   523→   523→{
   524→   524→  "request_id": "req_062",
   525→   525→  "response_id": "res_xxxxxxxx",
   526→   526→  "type": "search_jump_result",
   527→   527→  "keyword": "ERROR",
   528→   528→  "occurrence": 50,
   529→   529→  "total_occurrences": 47,
   530→   530→  "found": false,
   531→   531→  "output": "",
   532→   532→  "output_truncated": false
   533→   533→}
   534→   534→```
   535→   535→
   536→   536→Sama seperti tidak ditemukan — `found: false`. AI tahu dari `total_occurrences` bahwa ada 47 kejadian, tapi occurrence ke-50 tidak ada.
   537→   537→
   538→   538→#### 5B.6 Error Cases
   539→   539→
   540→   540→| Kondisi | Error Code |
   541→   541→|---|---|
   542→   542→| `response_id` sudah di-evict dari ring buffer | `ERR_RESPONSE_EXPIRED` |
   543→   543→| `keyword` kosong string | `ERR_INVALID_REQUEST` |
   544→   544→| `occurrence` = 0 | `ERR_INVALID_REQUEST` (occurrence 1-based) |
   545→   545→
   546→   546→#### 5B.7 Implementasi: `visibox_search.c`
   547→   547→
   548→   548→```c
   549→   549→// visibox_search.c
   550→   550→
   551→   551→// Hasil pencarian
   552→   552→typedef struct {
   553→   553→    bool found;
   554→   554→    size_t found_line;          // 1-based, nomor baris absolut
   555→   555→    size_t total_occurrences;
   556→   556→} SearchResult;
   557→   557→
   558→   558→// Cari keyword di seluruh output buffer.
   559→   559→// Mengembalikan SearchResult yang berisi posisi occurrence ke-N.
   560→   560→// Pencarian dilakukan di raw buffer (tanpa line numbers).
   561→   561→SearchResult visibox_search_keyword(const OutputBuffer *buf,
   562→   562→                                     const char *keyword,
   563→   563→                                     bool case_sensitive,
   564→   564→                                     int occurrence);
   565→   565→
   566→   566→// Ambil baris-baris sekitar found_line sebagai string.
   567→   567→// line_start dan line_end di-set ke range absolut yang diambil.
   568→   568→// Jika line_numbers=true, output sudah diformat dengan line numbers.
   569→   569→// Mengembalikan string baru, caller free().
   570→   570→char *visibox_search_get_context(const OutputBuffer *buf,
   571→   571→                                  size_t found_line,
   572→   572→                                  int context_lines,
   573→   573→                                  bool line_numbers,
   574→   574→                                  size_t output_limit,
   575→   575→                                  size_t *out_line_start,
   576→   576→                                  size_t *out_line_end);
   577→   577→```
   578→   578→
   579→   579→**Algoritma pencarian:**
   580→   580→
   581→   581→```
   582→   582→1. Akses raw buffer dari OutputBuffer (ter-index per baris)
   583→   583→2. Iterasi semua baris, lakukan substring match (case-sensitive atau tidak)
   584→   584→3. Setiap match, increment counter
   585→   585→4. Ketika counter == occurrence (atau -1 untuk last), catat found_line
   586→   586→5. Return SearchResult
   587→   587→```
   588→   588→
   589→   589→**Kompleksitas:** O(N × M) dimana N = jumlah baris, M = panjang keyword. Untuk output 100K baris dengan keyword pendek, ini berjalan dalam hitungan milidetik. Tidak perlu index khusus — linear scan cukup karena pencarian hanya dilakukan sekali per request, bukan di hot path.
   590→   590→
   591→   591→**Optimasi (jika diperlukan di kemudian hari, bukan di Fase 1):**
   592→   592→- Boyer-Moore atau memchr untuk keyword pendek
   593→   593→- Line index (array of offsets) yang sudah ada di OutputBuffer — akses baris ke-N adalah O(1)
   594→   594→
   595→   595→#### 5B.8 Hubungan `search_jump` ↔ `fetch_page`
   596→   596→
   597→   597→`search_jump` dan `fetch_page` saling melengkapi:
   598→   598→
   599→   599→```
   600→   600→AI menerima output 10.000 baris (truncated)
   601→   601→  ↓
   602→   602→AI ingin cari "ERROR" → search_jump
   603→   603→  ↓
   604→   604→Response: found_line=1842, page_hint=37, cursor=cur_xxx
   605→   605→  ↓
   606→   606→AI ingin lihat lebih banyak konteks → fetch_page(cursor=cur_xxx)
   607→   607→  ↓
   608→   608→AI mendapat halaman 37 (baris 1801-1900) dengan line_numbers
   609→   609→  ↓
   610→   610→AI merujuk baris spesifik: "baris 1842 menunjukkan ERROR pada replica-3"
   611→   611→```
   612→   612→
   613→   613→Flow ini memungkinkan AI:
   614→   614→1. Cepat menemukan apa yang dicari tanpa scroll manual
   615→   615→2. Loncat ke halaman terkait untuk analisis lebih dalam
   616→   616→3. Merujuk baris spesifik dengan nomor absolut yang unik
   617→   617→

   620→   620→## 6. PROTOKOL — KONTRAK LENGKAP
   621→   621→
   622→   622→Setiap request type di bawah punya: field wajib, field opsional dengan default, dan daftar error yang mungkin. Ini supaya tidak ada celah "AI kirim X, VisiBox tidak tahu harus jawab apa".
   623→   623→
   624→   624→### 6.0 Error format standar (dipakai semua request type)
   625→   625→
   626→   626→```json
   627→   627→{
   628→   628→  "request_id": "req_xxx",
   629→   629→  "response_id": "res_xxxxxxxx",
   630→   630→  "type": "error",
   631→   631→  "error_code": "ERR_SESSION_NOT_FOUND",
   632→   632→  "error_message": "Session sess_xxx tidak ditemukan atau sudah ditutup",
   633→   633→  "details": {}
   634→   634→}
   635→   635→```
   636→   636→
   637→   637→Daftar kode error baku (dipakai konsisten, bukan string bebas per handler):
   638→   638→
   639→   639→| Kode | Arti |
   640→   640→|---|---|
   641→   641→| `ERR_INVALID_REQUEST` | JSON tidak valid atau field wajib hilang |
   642→   642→| `ERR_SESSION_NOT_FOUND` | `session_id` tidak ada di registry (sudah ditutup/expired/salah) |
   643→   643→| `ERR_SESSION_LIMIT_REACHED` | Sudah mencapai `max_sessions`, harus tutup salah satu dulu |
   644→   644→| `ERR_RESPONSE_EXPIRED` | `reference_response_id` sudah di-evict dari ring buffer |
   645→   645→| `ERR_CURSOR_INVALID` | Cursor tidak dikenal atau sudah lewat `cursor_expiry_ms` |
   646→   646→| `ERR_TIMEOUT` | Operasi melewati `timeout_ms` tanpa hasil (bukan error eksekusi, hanya timeout) |
   647→   647→| `ERR_EXEC_FAILED` | Gagal spawn proses (PTY alloc gagal, command tidak ditemukan, dll) |
   648→   648→| `ERR_SEARCH_KEYWORD_EMPTY` | `search_jump` dengan keyword kosong (v3) |
   649→   649→| `ERR_INTERNAL` | Bug internal VisiBox, sertakan detail untuk debug |
   650→   650→

   695→   695→**Response tanpa line numbers** (default, backward compatible):
   696→   696→```json
   697→   697→{
   698→   698→  "request_id": "req_001",
   699→   699→  "response_id": "res_xxxxxxxx",
   700→   700→  "type": "execute_result",
   701→   701→  "exit_code": 0,
   702→   702→  "output": "Filesystem      Size  Used Avail Use% Mounted on\n/dev/sda1       50G   28G   20G  59% /",
   703→   703→  "output_lines": 8,

   716→   716→  "type": "execute_result",

   725→   725→### 6.2 `session_start`
   726→   726→
   727→   727→**Request:**
   728→   728→```json
   729→   729→{
   730→   730→  "request_id": "req_010",
   731→   731→  "type": "session_start",
   732→   732→  "command": "ssh user@192.168.1.100",
   733→   733→  "options": {
   734→   734→    "env": {"TERM": "xterm-256color"},
   735→   735→    "cwd": "/home/user",

   737→   737→    "output_limit": 100

   741→   741→

Kalau gagal alloc PTY atau `MAX_SESSIONS` tercapai → `ERR_SESSION_LIMIT_REACHED` atau `ERR_EXEC_FAILED`, tidak ada session yang teregister.

   744→   744→### 6.3 `session_input`
   745→   745→
   746→   746→```json
   747→   747→{
   748→   748→  "request_id": "req_011",
   749→   749→  "type": "session_input",
   750→   750→  "session_id": "sess_9d8e7f6a",
   751→   751→  "input": "mypassword\n",
   752→   752→  "options": {
   753→   753→    "wait_for_prompt": true,
   754→   754→    "prompt_pattern": "\\$\\s*$",
   755→   755→    "timeout_ms": 10000,
   756→   756→    "output_limit": 100,
   757→   757→    "line_numbers": true
   758→   758→  }
   759→   759→}
   760→   760→```
   761→   761→
   762→   762→`prompt_pattern` dicocokkan terhadap output yang sudah di-strip ANSI escape sequence (lewat `visibox_prompt.c`). Field `output` di response tetap berisi raw output (termasuk ANSI). `line_numbers` *[BARU v3]* bekerja sama seperti di `execute`.
   763→   763→
   764→   764→Kalau `session_id` tidak ditemukan → `ERR_SESSION_NOT_FOUND`, tidak ada side effect.
   765→   765→
   766→   766→### 6.4 `session_read`
   767→   767→
   768→   768→Sama seperti v1 — baca tanpa kirim input, untuk polling output baru (misal proses background di session yang sedang jalan lama). Mendukung `line_numbers` *[BARU v3]*.
   769→   769→
   770→   770→### 6.5 `session_list`
   771→   771→
   772→   772→Sama seperti v1, tambah field `uptime_ms` per session untuk kemudahan AI menilai session mana yang sudah lama idle dan layak ditutup.
   773→   773→
   774→   774→### 6.6 `session_close`
   775→   775→
   776→   776→Sama seperti v1. Tambahan: kalau `session_id` sudah closed sebelumnya (dipanggil dua kali), response bukan error — kembalikan status `already_closed: true` dengan `final_output` kosong, supaya AI yang retry karena network flaky tidak salah anggap gagal.
   777→   777→
   778→   778→### 6.7 `fetch_page` / `session_fetch_page`
   779→   779→
   780→   780→```json
   781→   781→{
   782→   782→  "request_id": "req_051",
   783→   783→  "type": "fetch_page",
   784→   784→  "response_id": "res_page01",
   785→   785→  "cursor": "cur_8f7e6d5c4b3a",
   786→   786→  "options": {
   787→   787→    "output_limit": 50,
   788→   788→    "line_numbers": true
   789→   789→  }
   790→   790→}
   791→   791→```
   792→   792→
   793→   793→Kalau `response_id` sudah di-evict dari ring buffer → `ERR_RESPONSE_EXPIRED`. Kalau cursor valid formatnya tapi sudah lewat `cursor_expiry_ms` → `ERR_CURSOR_INVALID`.
   794→   794→
   795→   795→### 6.8 `search_jump` *(BARU v3)*
   796→   796→
   797→   797→Lihat detail lengkap di §5B.2 dan §5B.3.
   798→   798→
   799→   799→Ringkasan kontrak:
   800→   800→- **Wajib:** `response_id`, `keyword`
   801→   801→- **Opsional:** `options.case_sensitive`, `options.occurrence`, `options.context_lines`, `options.output_limit`, `options.line_numbers`
   802→   802→- **Error:** `ERR_RESPONSE_EXPIRED`, `ERR_INVALID_REQUEST` (keyword kosong / occurrence=0)
   803→   803→- **Bukan error:** `found: false` — keyword memang tidak ada, AI lanjut coba keyword lain

   807→   807→## 7. DATA STRUCTURES
   808→   808→
   809→   809→### 7.1 Dari v2 (tidak berubah, diringkas)
   810→   810→
   811→   811→```c
   812→   812→// ═══════════════════════════════════════
   813→   813→// RESPONSE STORE — bounded ring buffer
   814→   814→// ═══════════════════════════════════════
   815→   815→typedef struct {
   816→   816→    char response_id[16];
   817→   817→    OutputBuffer *buffer;      // pointer, BUKAN copy — owned oleh store
   818→   818→    time_t created_at;
   819→   819→    size_t approx_bytes;       // untuk hitung total_bytes_used di bawah
   820→   820→} StoreEntry;
   821→   821→
   822→   822→typedef struct {
   823→   823→    StoreEntry entries[VISIBOX_STORE_MAX_ENTRIES];  // dari config, default 256
   824→   824→    size_t head;                 // posisi FIFO write berikutnya
   825→   825→    size_t count;
   826→   826→    size_t total_bytes_used;     // dijaga <= store_max_bytes dari config
   827→   827→    pthread_mutex_t lock;
   828→   828→} ResponseStore;
   829→   829→
   830→   830→// Eviction: FIFO ketat. Saat entry baru ditambah dan
   831→   831→// (count == MAX_ENTRIES) ATAU (total_bytes_used + new_entry_bytes > store_max_bytes),
   832→   832→// entry tertua (head) di-evict dulu sebelum insert.
   833→   833→// fetch_page/search_jump terhadap response_id yang sudah di-evict → ERR_RESPONSE_EXPIRED.
   834→   834→
   835→   835→// ═══════════════════════════════════════
   836→   836→// EVENT LOOP — satu loop untuk semua session
   837→   837→// ═══════════════════════════════════════
   838→   838→typedef struct {
   839→   839→    int epoll_fd;
   840→   840→    struct epoll_event events[MAX_SESSIONS];
   841→   841→    Session *fd_to_session[VISIBOX_MAX_FD];
   842→   842→} EventLoop;
   843→   843→
   844→   844→// ═══════════════════════════════════════
   845→   845→// PROMPT DETECTION dengan ANSI strip
   846→   846→// ═══════════════════════════════════════
   847→   847→char *visibox_strip_ansi(const char *raw, size_t len);
   848→   848→bool visibox_detect_prompt(const OutputBuffer *buf, const char *pattern);
   849→   849→```
   850→   850→
   851→   851→### 7.2 Baru di v3
   852→   852→
   853→   853→```c
   854→   854→// ═══════════════════════════════════════
   855→   855→// LINE NUMBERS (visibox_linenums.c)
   856→   856→// ═══════════════════════════════════════
   857→   857→
   858→   858→// Format separator line number (default: " │ ")
   859→   859→#define VISIBOX_LINENUM_SEP_DEFAULT  " \xe2\x94\x82 "  // UTF-8: │ (U+2502)
   860→   860→
   861→   861→typedef struct {
   862→   862→    char separator[16];         // default: " │ "
   863→   863→    bool enabled;               // global default dari config
   864→   864→} LineNumberConfig;
   865→   865→
   866→   866→// ═══════════════════════════════════════
   867→   867→// SEARCH (visibox_search.c)
   868→   868→// ═══════════════════════════════════════
   869→   869→
   870→   870→typedef struct {
   871→   871→    bool found;
   872→   872→    size_t found_line;          // 1-based, nomor baris absolut di seluruh output
   873→   873→    size_t total_occurrences;   // total baris yang mengandung keyword
   874→   874→} VisiboxSearchResult;
   875→   875→
   876→   876→// ═══════════════════════════════════════
   877→   877→// OUTPUT BUFFER — ditambah field untuk line index
   878→   878→// ═══════════════════════════════════════
   879→   879→// OutputBuffer dari v2 sudah menyimpan data per-baris.
   880→   880→// Untuk v3, ditambahkan line index (array of {offset, length} per baris)
   881→   881→// supaya akses baris ke-N adalah O(1) — penting untuk search_jump.
   882→   882→
   883→   883→typedef struct {
   884→   884→    size_t offset;    // byte offset di buffer utama
   885→   885→    size_t length;    // panjang baris (termasuk \n kalau ada)
   886→   886→} LineEntry;
   887→   887→
   888→   888→// Di dalam OutputBuffer:
   889→   889→//   LineEntry *lines;         // array, di-alloc setelah semua data terkumpul
   890→   890→//   size_t line_count;        // = total_lines
   891→   891→//   bool line_index_built;    // lazily built saat pertama kali search_jump/fetch_page dipanggil
   892→   892→```
   893→   893→
   894→   894→**Mengapa line index perlu di-build lazily:** Saat drain pipe, kita tidak tahu berapa total baris sampai selesai. Line index di-build setelah drain selesai, sebelum buffer dimasukkan ke response store. Ini O(N) sekali, lalu setiap akses baris ke-N adalah O(1).
   895→   895→
   896→   896→### 7.3 Response struct — ditambah field v3
   897→   897→

   899→   899→// VisiboxResponse — field baru v3
   900→   900→typedef struct {
   901→   901→    // ... (field v2 tidak berubah) ...
   902→   902→
   903→   903→    // [BARU v3] Line number metadata
   904→   904→    bool line_numbers;          // apakah output saat ini berisi line numbers
   905→   905→    size_t line_start;          // nomor baris pertama di output ini (absolut)
   906→   906→    size_t line_end;            // nomor baris terakhir di output ini (absolut)
   907→   907→} VisiboxResponse;
   908→   908→```
